== Convert instruction type fields (SrcType/DstType)

The Convert instruction family uses two 5-bit type descriptor fields:

* ``SrcType``: describes the *source* data type.
* ``DstType``: describes the *destination* data type.

Both fields are interpreted according to the instruction mnemonic:

* ``V.FCVT``: ``SrcType`` and ``DstType`` use the *floating-point* type table.
* ``V.FCVTI``: ``SrcType`` uses the *floating-point* type table; ``DstType`` uses the *integer* type table.
* ``V.ICVT``: ``SrcType`` and ``DstType`` use the *integer* type table.
* ``V.ICVTF``: ``SrcType`` uses the *integer* type table; ``DstType`` uses the *floating-point* type table.

Illegal encodings::
+
+If either ``SrcType`` or ``DstType`` is a reserved encoding for the relevant table, the instruction encoding is *illegal*.
Software must not generate illegal encodings.

NOTE: Convert instructions are defined as *value conversions* (numeric conversions), not bit reinterpretations.

NOTE: For the current `v0.4` bring-up profile, these explicit `SrcType` / `DstType`
fields are the only canonical encoded vector type selectors. Non-convert `v.*`
forms do not obtain type information from `RegId10`; instead they use
mnemonic-defined default decoded type shadows over the 32-bit vector carrier
file (`u32` or `s32` for non-convert integer families, and the agreed default
carrier-domain FP rule for non-convert `V.F*` families).

=== Unified conversion intent (staged)

* If destination width > source width: widening conversion.
* If destination width < source width: narrowing conversion.
* Integer↔integer conversions use *numeric* semantics (sign/zero extension and truncation), not bit reinterpretation.
* Float↔integer and float↔float special cases (NaN/Inf/Subnormal; rounding and exception flags) are defined by the FSU/CSTATE rules.
