# LinxISA v0.57 PTO Block Encoding Map

This page defines the PTO tile/data instruction forms that receive Linx block encodings under the current LinxISA v0.57 block definition. Each encoded instruction is represented as one block record: `BSTART.*`, the required descriptors, optional body records when the block class needs them, and `BSTOP`.

Synchronization, communication, pipe lifecycle, pipe-synchronization, debug, and IR-only virtual/view operations are not PTO tile/data instruction encodings. They MUST NOT appear in the opcode rows below; they either lower to the tile/data instructions listed here or remain compiler/runtime constructs.

## Encoding Rules

- Every PTO instruction form listed in this map is represented by one Linx block record: BSTART.*, descriptors, optional body records, BSTOP.
- Multiple PTO spellings may share one encoding row when they differ only by form bits, operand presence, or mode fields.
- Tile operands use B.ITP for sources and B.OTA for destinations.
- Element-domain shape, valid-region, and mask state use B.META.
- Matrix dimensions use B.DIM only when the opcode profile requires M/N/K.
- Scalar, memory, dependency, data-attribute, and configuration operands use B.IOR, B.IOD, B.DATR, and B.ARG according to descriptor ownership.

Descriptor ownership is fixed by the v0.57 block definition: tile sources use `B.ITP`, tile destinations use `B.OTA`, tile shape metadata uses `B.META`, matrix dimensions use `B.DIM`, scalar or memory operands use `B.IOR`, dependency operands use `B.IOD`, data attributes use `B.DATR`, and mode/layout/configuration fields use `B.ARG`.

## Encoded Opcode Spaces

| Space | Use |
| --- | --- |
| `TEPL.TileOpcode` | Elementwise, tile-scalar, reduce/expand, partition, sort, random, and tile-complex instructions. |
| `CUBE.Function` | Matrix and GEMV instructions, with suffix variants selected by B.ARG form fields and operand presence. |
| `TMA.Function` | Load, store, gather, scatter, and prefetch instructions. |
| `FIXP.Function` | Move, convert, quantize, insert, extract, concat, fill-pad, transpose, and related fixed-pipeline instructions. |

## Excluded Non-Instruction Surfaces

| Surface | Rule |
| --- | --- |
| Synchronization, event, flag, barrier, and buffer-token APIs | Scheduling/runtime ordering constructs only. Do not allocate PTO block opcodes. |
| Communication APIs | Runtime or fabric services only. Do not allocate PTO tile/data block opcodes. |
| Pipe lifecycle and pipe-synchronization APIs | Pipe-handle allocation/transfer/synchronization only. Do not allocate PTO block opcodes. |
| IR-only virtual, view, metadata, and debug operations | IR lowering or debug constructs only. Do not allocate PTO block opcodes. |

## Merge Policy

- Suffix forms such as `_FP`, `_ASYNC`, `.ACC`, `.BIAS`, and `.MX` share an opcode key when the difference can be represented by `B.ARG` form bits, operand presence, or data-attribute fields.
- Merged rows still list every PTO spelling they cover; the row count is the binary encoding count, not the API spelling count.
- Optional tmp operands are explicit operands. A tmp tile uses `B.ITP`; a tmp scalar or pointer uses `B.IOR`.
