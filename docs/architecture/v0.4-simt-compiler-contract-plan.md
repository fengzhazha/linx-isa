# LinxISA v0.4 SIMT Compiler Contract Plan

Status: Planning page
Last updated: 2026-03-09

This page defines the work needed to turn the current `v0.4` SIMT description
into a compiler-grade contract. It is not itself normative. Its job is to
identify which architecture details must be frozen so LLVM, QEMU, RTL, and
alternative compilers stop relying on private conventions.

Direction now frozen in canonical `v0.4`:

- SIMT kernels are **group-granular divergent kernels**,
- divergence and reconvergence are expressed through explicit `p` management,
- group-internal branches are legal within the `B.TEXT` body region,
- canonical `v0.4` does not require a hidden architectural per-lane PC stack.

## Why This Plan Exists

Current `v0.4` canonical docs already freeze the high-level SIMT model:

- `VEC` is the general programmable SIMT engine.
- `MPAR` and `MSEQ` are the canonical shader-kernel block types.
- `B.TEXT` selects an out-of-line body.
- `p` is the architectural EXEC mask.

That is enough for the current replay-oriented bring-up subset. It is not yet a
complete compiler contract for branch-heavy SIMT kernels like the alternative
hash-table example analyzed in
`docs/architecture/research/linx-simt-alt-compiler-gap-analysis.md`.

The remaining work is to freeze the missing parts of the model so all compiler
and execution implementations share the same answers.

## Existing Baseline To Preserve

This plan must preserve the current canonical `v0.4` posture:

- `docs/architecture/v0.4-architecture-contract.md`

- `docs/architecture/v0.4-hardening-policy.md`

- `docs/architecture/isa-manual/src/chapters/04_block_isa.adoc`

In particular:

- `VEC` remains the general parallel-loop engine.
- `MPAR/MSEQ` remain the canonical programmable shader-kernel forms.
- `B.TEXT` remains the body entry mechanism for decoupled SIMT blocks.
- `p` remains distinct from block-control predicate state.

## Contract Workstreams

### Launch Geometry Contract

Freeze the exact relationship among:

- header mode (`VS8` / `VS16` and future width tags),
- logical thread count,
- `LB0/LB1/LB2`,
- `lc0/lc1/lc2`,
- lane count versus group count,
- strip-mined launch versus direct launch.

Required outputs:

- one canonical 1-D lowering rule,
- one optional grouped/strip-mined rule if supported,
- one statement on whether `LB0` may ever mean total logical threads.

Acceptance rule:

- two different compilers targeting canonical `v0.4` must derive the same
  logical lane index and group index from the same header.

### EXEC / Divergence Contract

Freeze whether canonical `v0.4` supports:

- predication-only SIMT control flow, or
- architectural divergence and reconvergence.

Status:

- Frozen to architectural divergence and reconvergence through explicit group PC
  plus EXEC-mask (`p`) management.

If predication-only:

- explicitly ban compiler-generated code shapes that require hidden
  reconvergence state.

If divergence is architectural:

- define the minimum required model:
  - lane-subset control,
  - reconvergence point semantics,
  - legal branch behavior under partial lane activity,
  - any required EXEC stack or save/restore semantics.

Acceptance rule:

- the manual must answer what happens when only some active lanes take an
  in-body branch.

### Body Control-Flow Contract

Extend the current `B.TEXT` body rules from entry/termination into a real body
control-flow contract.

Freeze:

- what an in-body `b.*` or `j` means in a SIMT body,
- whether branch decisions are scalar-uniform only,
- whether branch-on-mask is first-class or compiler-synthesized,
- how loop exits with partially completed lanes are represented.

Acceptance rule:

- the ISA manual must define inner-body branch semantics strongly enough that
  QEMU and RTL do not need compiler-specific assumptions.

### Lane-Local Scratch And Spill ABI

Freeze how canonical compilers use `.local` memory for SIMT-private state.

Define:

- which tile base is canonical compiler scratch,
- lane-private addressing formula,
- scratch lifetime and visibility,
- alignment and minimum guaranteed size,
- alias rules between compiler scratch and user-authored `.local` accesses.

Acceptance rule:

- a compiler-generated `.local` spill/reload pattern is architecturally
  meaningful without referring to one backend implementation.

### Operand-Domain Crossing Contract

Freeze the sanctioned ways that values move among:

- scalar/group control state,
- vector lane state,
- EXEC mask state.

Define whether canonical `v0.4` needs new first-class operations for:

- mask query (`any` / `all` / `none`),
- scalar-to-mask broadcast,
- mask save/restore,
- lane-state merge support beyond generic `V.PSEL`.

Acceptance rule:

- control-heavy SIMT lowering should no longer depend on unstructured
  `movr`/`ori` shuttle idioms as a de facto ABI.

## Promotion Order

These decisions should be frozen in this order:

1. Launch geometry
2. EXEC / divergence model
3. Body control-flow semantics
4. Scratch ABI
5. Operand-domain crossing rules
6. Compiler-facing lowering subset

This order is intentional. Compiler and emulator work should not outrun the
architecture decisions above them.

## Conflict Triggers

Implementation work should stop and escalate if any of these remain unresolved:

- one compiler treats `LB0` as lane count while another treats it as total
  threads,
- one execution model assumes predication-only SIMT while another assumes
  hidden reconvergence,
- `.local` scratch usage depends on backend-private layout or lifetime rules,
- QEMU/RTL correctness depends on a branch semantics rule not written in the
  manual.

## Exit Criteria

This plan is done when:

- the canonical manual and contract pages answer the open SIMT questions from
  the gap analysis,
- the compiler can target a written SIMT contract instead of a bring-up
  convention,
- QEMU and RTL can be validated against architectural semantics rather than
  backend-specific code shape.
