# 灵犀指令集 v0.5 指令变动说明（草案）

> 状态：Draft / 非现行规范
> 基线版本：LinxISA v0.4
> 基线来源：
> - `isa/v0.4/registers/tile_reg.json`
> - `docs/architecture/isa-manual/src/chapters/03_programming_model.adoc`
> - `docs/architecture/isa-manual/src/chapters/04_block_isa.adoc`
> - `docs/architecture/isa-manual/src/chapters/08_tile_blocks.adoc`
> - `docs/architecture/isa-manual/src/chapters/09_memory_operations.adoc`

本文描述面向 LinxISA v0.5 的指令与 Tile 架构变动方向。当前仓库的公开规范仍以 v0.4 为准；本文用于说明 v0.5 相对 v0.4 的计划性变动、兼容性影响以及需要冻结的新增架构对象。

## 文档目标

- 说明 v0.5 相对 v0.4 的 ISA 变化，不复述 v0.4 全量手册。
- 对 Tile、向量、张量与内存访问相关的变化给出统一口径。
- 把新增名词、约束、异常和兼容策略写清楚，便于后续同步到 LLVM、QEMU、RTL 和 ISA 手册。

## 变动总览

| 主题 | v0.4 基线 | v0.5 方向 |
| --- | --- | --- |
| Tile 寄存器规模 | 32 个 Tile，4 hand × 8 depth，总计 128KB 架构容量 | 扩展到 64 个 Tile，4 hand × 16 depth；容量管理改为动态分配 |
| Tile 容量模型 | 每个 Tile 固定受 512B..4KB 约束 | 单 Tile 动态 512B..256KB，总活跃容量受 512KB 上限约束 |
| Tile 架构状态 | 以“寄存器个数 × 单寄存器最大容量”建模 | 改为由 `TileInfo` / `LCPR` 描述的浮动容量模型 |
| Persistent Tile | 无 | 新增常驻核内 Tile 区域，跨块保活，关闭 rename 侵占 |
| Block 描述符 | `B.IOT` + `B.IOTI` 并存，`B.IOD` 仍保留为兼容项 | 删除 `B.IOD`，融合 `B.IOT/B.IOTI`，统一为立即数尺寸分配 |
| Predicate | 仅有块控制谓词和 kernel `p` | 新增 `PredicateTile`，支持 Tile 级 1bit 掩码 |
| Global/Bridge 命名 | kernel 全局内存走 `*.brg` | 统一重命名为 `*.global` |
| 低精度类型 | `INT4/UINT4` 与 4-bit float 基础类型 | 新增 `int4x2/uint4x2/e2m1x2/e1m2x2/hif4x2` |
| CUBE 命名 | `TMATMUL` 为规范名，`MAMULB*` 为兼容名 | 统一收敛到 `MATMUL*` 命名 |

## 总体兼容原则

- v0.5 对外是相对 v0.4 的不兼容升级，不能假设二进制编码自动兼容。
- v0.4 文档中的 `*.brg`、`TMATMUL*`、`B.IOD`、`B.IOT/B.IOTI` 双描述符模型，在 v0.5 中都需要显式迁移。
- 本文沿用 v0.4 当前的 1-based Tile 引用记法，即 `t#1` 表示 hand 内最近一次定义；扩展后使用 `t#1..t#16`、`u#1..u#16`、`m#1..m#16`、`n#1..n#16`。

## Tile 寄存器数量扩展到 64 个

### v0.4 基线

v0.4 规定 Tile 寄存器为 32 个，组织方式为 `T/U/M/N` 四个 hand，每个 hand 深度为 8。手册中的 Tile 引用是 hand-local queue 语义，`#1` 表示最近定义，`#8` 表示最老定义。

### v0.5 变动

v0.5 将 Tile 寄存器总数从 32 个增加到 64 个，方式为将四个 hand 的深度从 8 扩展到 16：

- `T#1 .. T#16`
- `U#1 .. U#16`
- `M#1 .. M#16`
- `N#1 .. N#16`

这样可以容纳更深的 tile 数据流、更多的 loop unroll 结果以及更复杂的 tile 依赖链。

### 输入生命周期标记

v0.5 继续保留 `.reuse`，并把 `.kill` 作为默认生命周期语义：

- `SrcTile.reuse` 表示该输入在当前块后仍保留可复用性。
- 未显式标注时，默认等价于 `SrcTile.kill`。

这意味着编译器在资源紧张时可以更积极地回收 Tile 容量，而在显式复用路径上保留常驻 tile。

### 编码方向

由于 Tile 索引具有局部性，v0.5 对近端和远端引用采用分层编码策略：

- 近端索引优先复用主 `B.IOT` 编码路径。
- 远端索引（超出近端窗口，例如 `#9..#16`）通过扩展描述符路径编码。

本文建议保留如下草案风格：

```asm
B.IOT [t#1], ->T<4KB>
C.B.IOTI [t#15], last ->T<4KB>
```

说明：

- `C.B.IOTI` 在 v0.5 中作为远端索引或压缩扩展描述符的候选记法；
- 具体位分配和压缩条件需要在 `isa/v0.5/opcodes/` 中冻结。

## Tile 架构状态重构：新增 `TileInfo` 与 `LCPR`

### v0.4 基线

v0.4 的公开模型是：

- 单 Tile 大小受 `512B..4KB` 约束；
- 32 个 Tile 的总架构容量为 `128KB`；
- `B.IOTI` 用立即数给出目标 Tile 大小，`B.IOT` 可使用寄存器给出动态大小。

该模型实现简单，但把“寄存器个数”和“寄存器容量上限”静态绑定，导致状态模型偏刚性。

### v0.5 变动

v0.5 将 Tile 状态改为动态容量模型，不再用 `Tile 寄存器个数 × 单寄存器最大容量` 来描述架构状态。新的对象为：

- `LCPR`：LinxCore Capability Register。描述当前 LXCPU 线程可用的总 Tile 容量上限。
- `TileInfo`：描述每个 Tile 的当前容量、维度信息和分类属性。

本文定义的 v0.5 约束如下：

- 单 Tile 动态容量范围：`512B..256KB`
- 单线程活跃 Tile 总容量上限：`512KB`
- Tile 数据在物理存储上必须连续

### `TileInfo` 的逻辑字段

`TileInfo` 需要覆盖 64 个 Tile 的动态元数据，至少包括：

- 每个 Tile 的 `TileSize`
- 每个 Tile 的 `B.DIM` 参数快照：`LB0/LB1/LB2`
- 每个 Tile 的类别信息：`BARG`、分型、`PredicateTile` 标记、生命周期状态等

需要注意：单个 64-bit SSR 无法容纳 64 个 Tile 的全部元数据，因此 v0.5 更合理的实现方式不是“单寄存器”，而是“`TileInfo` SSR 家族”或“按 Tile 索引访问的 banked SSR 视图”。如果后续仍保留 `TileInfo` 这个单数名字，应在 `system_registers` 中显式定义它的索引化访问规则。

### 分配与异常

v0.5 中，新 Tile 的分配由统一后的 `B.IOT` 完成：

```asm
B.IOT [t#2.reuse], ->T<64KB>
```

执行该指令时，硬件必须进行容量检查：

- 若分配后所有活跃 Tile 的总容量不超过 `LCPR.TileCapacity`，则分配成功；
- 若超过上限，则必须产生 `TileRegisterOverflow` 异常。

建议把 `TileRegisterOverflow` 作为新的 block/tile 类异常纳入 v0.5 trap 编码，而不是复用 v0.4 中已有但语义更宽泛的 `EC_BLOCKFMT`。

### 编译器约束

编译器在 v0.5 中必须同时管理两个资源：

- Tile 个数
- Tile 总容量

因此：

- 大 Tile 会减少可并发存活的 Tile 数量；
- 深度 unroll 会增加 hand 压力和总容量压力；
- `kill` / `reuse` 将从“提示位”提升为影响容量驻留策略的重要输入。

## 新增 Persistent Tile

### 目标

v0.5 引入 Persistent Tile，目的是让软件获得类似核内 buffer/shared memory 的交互方式：一块显式保留的本地 Tile 空间，可跨块存活，并且不再被普通 Tile rename 机制侵占。

### 架构语义

Persistent Tile 的属性如下：

- 跨不同块指令存活；
- 普通块内的 Tile rename 不能覆盖这片区域；
- 地址空间从 `0` 开始，最大范围为 `512KB`；
- Persistent Tile 容量应计入 `LCPR` 报告的总 Tile 容量预算，而不是额外赠送一块独立状态。

### 分配与释放

Persistent Tile 通过 `B.IOT` 管理：

```asm
B.IOT [], ->PT<64KB>
```

释放方式为：

```asm
B.IOT [], ->PT.FREE
```

### 访问约束

- 只有 `VPAR`、`VSEQ`、`MPAR`、`MSEQ` 四类块允许访问 Persistent Tile。
- 标量访问使用 `l.load.local` / `l.store.local`。
- 向量访问使用 `v.load.local` / `v.store.local`。
- 访问地址不得越过已分配的 Persistent Tile 范围。

本文建议在 `.local` 操作数空间中新增 `PT` base，例如：

```asm
l.load.local [PT, 0x40], ->p
v.store.local vt#1, [PT, lc0<<2, 0x100]
```

### 保序规则

当块头部带有 `PT` 访问标记时，v0.5 对 Persistent Tile 访问采用最老块优先的保序模型：

- 同时存在多个 `PT` 块时，只有最老的 `PT` 块可以访问 Persistent Tile；
- 该块提交后，Persistent Tile 的访问权移交给下一个最老的 `PT` 块。

这里的“`PT` 标记”需要在 v0.5 中落为一个明确的编码对象，建议放入块头属性位，而不是隐含在 block type 中。

## Block 描述符微调

### 删除 `B.IOD`

v0.4 中 `B.IOD` 已经处于“obsolete compatibility descriptor”状态。v0.5 彻底删除该指令，不再为 legacy 假依赖流保留编码语义。

### 融合 `B.IOT` 与 `B.IOTI`

v0.4 同时存在：

- `B.IOT`：目标 Tile 大小可由寄存器动态提供
- `B.IOTI`：目标 Tile 大小由立即数提供

v0.5 统一为单一 `B.IOT`，并做如下调整：

- 删除寄存器输入式目标大小；
- Tile 大小统一由立即数表达；
- 原 `B.IOTI` 的终结描述符语义、源 Tile 绑定语义和目标 Tile 类别语义并入新的 `B.IOT`。

这项调整与动态容量模型并不矛盾。v0.5 的“动态”体现在每次分配的 Tile 大小可不同，而不是运行时再通过 GPR 传入一个任意大小。

## 新增 `PredicateTile`

### v0.4 基线

v0.4 已有两类谓词：

- 块控制谓词 `BARG.CARG`
- kernel `EXEC` 谓词寄存器 `p`

但没有“Tile 自身携带 1bit 掩码”的类型。

### v0.5 变动

v0.5 新增 `PredicateTile`。它是带有 1bit 元素语义的 Tile 类型或 Tile 属性，用于描述高维 Tile 访问中的保护掩码。

本文建议：

- 在 `TileInfo` 中增加 `PredicateTile` 标记；
- `PredicateTile` 的每个逻辑元素为 1 bit；
- TMA/Tensor 指令可把 `PredicateTile` 作为额外 Tile 输入。

### 标量读取到 `p`

v0.5 允许标量 form 从 Tile 本地空间读取 1 bit，并装入谓词寄存器：

```asm
l.load.local [TA, 0], ->p
```

本文采用的默认解释是：读取出的 1 bit 被广播为 `p` 的全位值，即：

- 读到 `0`，则 `p` 全清零；
- 读到 `1`，则 `p` 全置一。

如果后续希望定义为“只写 `p[0]`”或“写 lane-local mask fragment”，需要另起 v0.5 子规范明确，不宜在主文中保持含糊。

### TMA 与稀疏访问

`TLOAD`、`TSTORE`、`MGATHER`、`MSCATTER` 在 v0.5 中支持额外的 `PredicateTile` 输入，其中每个 1 bit 作为对应元素的访问 mask：

- `1` 表示该元素参与访问；
- `0` 表示该元素被屏蔽。

这使得 Tile 级条件读写不再依赖外部标量循环拆解。

## `bridge` 与 `global` 指令命名合并

### v0.4 基线

当前公开 v0.4 规范中，kernel 全局内存访问的规范名字是：

- `l.*.brg`
- `v.*.brg`

`.local` 仍表示 Tile/local 方向访问。

### v0.5 变动

v0.5 删除 `bridge` 命名，统一使用 `global`：

- 删除标量 `l.load.brg` / `l.store.brg` 指令族名称
- 删除向量 `v.load.brg` / `v.store.brg` 指令族名称
- 统一为 `l.load.global` / `l.store.global`
- 统一为 `v.load.global` / `v.store.global`

这里的变化主要是命名和抽象层次的收敛：

- 对软件而言，访问的是 global memory，而不是“是否经过 bridge”这一微架构细节；
- 对实现而言，底层仍可沿用 v0.4 的 MTC/bridge 路径，只是 ISA 不再把该实现细节暴露为助记符。

`.local` 语义保持不变，并继续承担 Tile 与 Persistent Tile 的本地访问。

## 新增低精度数据类型

### v0.4 基线

v0.4 已有 4-bit 相关类型基础，但公开命名仍偏底层，例如 `INT4`、`UINT4` 以及若干 4-bit float 编码。

### v0.5 变动

为提升张量指令和数据搬运指令对低精度格式的表达能力，v0.5 新增以下面向软件的类型名：

- `int4x2`
- `uint4x2`
- `e2m1x2`
- `e1m2x2`
- `hif4x2`

这些类型首先用于：

- `CUBE` / `MATMUL` 类指令
- `TLOAD` 类指令

### 命名语义

`x2` 表示一个字节内打包两个 4-bit 元素。其软件语义是“两个逻辑元素的打包类型”，而不是单个 8-bit 标量。

因此在 v0.5 中需要统一以下规则：

- 元素计数按逻辑元素数计算，而不是按物理字节数计算；
- `TileSize` 仍按字节计量；
- `LB0/LB1/LB2` 的矩阵维度语义按逻辑元素解释。

## CUBE 指令重命名为 `MATMUL` 家族

### v0.4 基线

当前仓库的 v0.4 公开规范里：

- 规范名是 `TMATMUL` / `TMATMUL.ACC`
- `MAMULB` / `MAMULB.ACC` 仅作为兼容拼写
- 输出物化另有 `ACCCVT`

### v0.5 变动

为了与 PTO 命名统一，v0.5 将 CUBE 家族的公开名字收敛为 `MATMUL`：

- `MAMULB` / `TMATMUL` -> `MATMUL`
- `MAMULB.ACC` / `TMATMUL.ACC` -> `MATMUL.ACC`
- `MAMULBAC` -> `MATMUL.BIAS`

### 兼容处理建议

由于当前仓库并不存在 `MAMULBAC` 的公开规范名，而是 `ACCCVT`，因此 v0.5 需要显式补一张别名和语义映射表：

- 哪些旧助记符只是纯重命名；
- 哪些旧助记符实际对应不同功能阶段；
- `MATMUL.BIAS` 是否完全替代 `ACCCVT`，还是仅覆盖其中一类 bias materialization 语义。

在这张映射表冻结之前，不建议把 `ACCCVT` 直接无条件翻译成 `MATMUL.BIAS`。

## 仍需冻结的关键问题

- `TileInfo` 是单 SSR 名称下的 banked family，还是一组显式编号的 SSR？
- `Persistent Tile` 的 `PT` 标记放在 `B.ATTR`、`BSTART`，还是新的 header descriptor 中？
- `TileRegisterOverflow` 的 trap 编码挂在哪个 major/minor cause 下？
- `PredicateTile` 是独立 Tile 类型，还是普通 Tile 上的属性位？
- `MATMUL.BIAS` 与现有 `ACCCVT` 的关系是“重命名”还是“新语义”？
- 远端 Tile 索引的压缩/扩展编码是否确定使用 `C.B.IOTI` 记法？

## 建议的规范定位

在 v0.5 尚未形成完整 `isa/v0.5` 目录前，本文应被视为：

- v0.5 指令与 Tile 架构变动说明
- 相对 v0.4 的迁移文档
- 后续正式 ISA 合同页和 manual patch 的输入草案

它不应替代当前 v0.4 的公开规范页，也不应被实现方误读为“已经冻结的现行定义”。
